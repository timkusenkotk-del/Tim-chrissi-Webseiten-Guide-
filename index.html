Kapitel 1

Bauen — der Einstieg

Der Kerngedanke ist einfach: HTML gibt einer Seite die Struktur, CSS das Aussehen, JavaScript das Verhalten. Wer darüber hinaus geht, wählt ein Framework wie React oder Vue für komplexe, interaktive Anwendungen — oder Astro, wenn viel statischer Content und Ladegeschwindigkeit im Vordergrund stehen. Für einfachere Projekte ist oft gar kein Framework nötig.

Empfohlener Pfad: HTML/CSS solide lernen → ein kleines Projekt bauen → dann ein Framework hinzunehmen, wenn es wirklich gebraucht wird.

Struktur

HTML

Semantische Elemente wie <header>, <main>, <article> machen Inhalte für Mensch und Maschine verständlich.

Aussehen

CSS

Layout (Flexbox, Grid), Typografie und Farben. Mit Custom Properties baust du dir ein eigenes Design-System.

Verhalten

JavaScript

Interaktion, Daten, Animationen. Modernes JS reicht für sehr viel — Frameworks erst, wenn es wirklich nötig wird.

Kapitel 2

Beurteilen — die wichtigsten Maßstäbe

Eine gute Webseite erfüllt gleichzeitig mehrere Kriterien. Die wichtigsten lassen sich direkt mit Lighthouse prüfen — eingebaut in den Chrome-DevTools (F12 → Lighthouse). Lighthouse vergibt für jeden Bereich einen Score von 0–100. Ein gutes Ziel sind 90+ in allen vier Kategorien.

Speed
90+ /100

Performance

Lädt die Seite in unter 2–3 Sekunden? Wie ist der Largest Contentful Paint (LCP)?

A11y
90+ /100

Zugänglichkeit

Lässt sich alles mit der Tastatur bedienen? Reicht der Farbkontrast?

Search
90+ /100

SEO

Sinnvolle Überschriften, Alt-Texte für Bilder, lesbare URL-Struktur?

Trust
90+ /100

Sicherheit

Läuft die Seite über HTTPS? Gibt es offensichtliche Lücken?

Praktischer Tipp zum Starten

Den schnellsten Einstieg bieten CodePen (direkt im Browser ausprobieren) oder eine lokale Datei index.html in VS Code mit der Extension "Live Server". Für eigene Projekte ist Next.js + Vercel derzeit der verbreitetste Stack — kostenlos deploybar, sehr gute Dokumentation.

Kapitel 3

Design-Checkliste

Hak die Punkte ab, während du deine Seite anschaust — das geht auch gut mit dem Chrome-Inspektor nebenbei. Dein Fortschritt wird lokal im Browser gespeichert.

Fortschritt

0 / 0

Noch nichts abgehakt — leg los.

Typografie

Maximal 2 SchriftartenEine für Überschriften, eine für Fließtext — mehr wirkt unruhig.
Klare Hierarchie in SchriftgrößenH1 > H2 > H3 > Body — jede Ebene klar unterscheidbar.
Zeilenlänge zwischen 45–75 ZeichenZu breit = anstrengend zu lesen; zu schmal = zerhackter Lesefluss.
Zeilenabstand ca. 1.5× SchriftgrößeBody-Text bei 16px → line-height: 24px ist ein guter Ausgangspunkt.

Farbe

Farbpalette hat maximal 3–4 FarbenPrimär, Sekundär, Akzent, Neutral — mehr verwirrt.
Kontrastverhältnis ≥ 4.5:1 für TextWCAG AA. Tools: contrast-ratio.com oder Chrome DevTools → Farbwähler.
Farben haben klare BedeutungRot = Fehler, Grün = Erfolg — konsequent, nie umgekehrt.
Hintergrund und Oberflächen unterscheidbarCards, Modals und Seite sollten sich in Helligkeit klar unterscheiden.

Abstände & Raster

Abstände folgen einem festen Systemz. B. immer Vielfache von 4px oder 8px — niemals 13px, 17px, 22px.
Zusammengehöriges liegt näher beieinanderGesetz der Nähe: Label sitzt näher am Eingabefeld als an der Überschrift darüber.
Ausreichend WeißraumVollgepackte Seiten wirken billig — Luft ist kein verschwendeter Platz.

Komponenten & Konsistenz

Buttons sehen überall gleich ausGleiche Größe, Radius, Farbe — nicht mal rund, mal eckig.
Icons haben gleichen Stil und GrößeNicht mischen: outline-Icons neben filled-Icons; nicht 16px neben 24px.
Border-radius ist konsistentWenn Cards 12px Radius haben, sollten Buttons nicht 0px oder 999px haben.
Hover- und Focus-Zustände vorhandenJedes interaktive Element braucht visuelles Feedback — auch für Tastaturnutzer.

Schnell-Test

Squint-Test: Was sieht man zuerst?Augen zusammenkneifen und schauen — springt das Wichtigste ins Auge?
5-Sekunden-Test bestandenFremden zeigen, Timer starten — können sie danach sagen, worum es geht?
Ein paar Punkte verdienen besondere Aufmerksamkeit: Das Spacing-System ist der häufigste Fehler bei selbst gebauten Seiten — sobald Abstände nicht aus einer festen Skala kommen, wirkt alles leicht unruhig, ohne dass man genau sagen kann warum. Der Squint-Test am Ende ist ungewöhnlich, aber sehr effektiv: Augen zusammenkneifen und die Seite verschwimmen lassen. Was übrig bleibt, ist die tatsächliche visuelle Hierarchie.
Kapitel 4

Tools

VS Code ist ein kostenloser Code-Editor von Microsoft — heute der mit Abstand meistgenutzte Editor für Webentwicklung. Schlank, schnell, durch Extensions auf jede Sprache und jeden Workflow zugeschnitten.

Mein empfohlener Einstieg: Prettier und Live Server reichen für die ersten Wochen komplett aus. Prettier einmal konfigurieren (Format on Save aktivieren), Live Server per Rechtsklick auf die index.html starten — und du hast sofort eine angenehme Arbeitsumgebung ohne Ablenkung durch zu viele Tools.

VS Code Setting

Auto-Save bei Fokuswechsel

"files.autoSave": "onFocusChange" in Kombination mit Live Server bedeutet: der Browser aktualisiert sich, sobald du Alt+Tab drückst. Du musst nie mehr manuell Strg+S drücken.

VS Code Setting

Linked Editing für HTML-Tags

"editor.linkedEditing": true ersetzt die Auto-Rename-Tag Extension vollständig — öffnendes und schließendes Tag werden automatisch synchron umbenannt.

Shortcut

Strg+D — nächstes Vorkommen

Wort markieren → mehrfach Strg+D drücken → alle Stellen gleichzeitig bearbeiten. Der Shortcut, der am meisten Zeit spart und den die wenigsten Einsteiger kennen.

Setup-Tipp

Extension per Command Palette

Statt durch den Marketplace zu klicken: Command Palette öffnen und z. B. ext install esbenp.prettier-vscode eingeben.

Claude.ai vs. Claude Code für Design

Claude Code lädt für Frontend-Arbeit automatisch einen Frontend-Design-Skill, der auf produktionsreifen Code mit distinktiver Ästhetik ausgerichtet ist. Du beschreibst, was du willst — Claude setzt es direkt als echten Code um, kein Figma-Export, kein Handoff. Ohne Anleitung tendiert Claude allerdings zu generischen Designs (Inter-Schrift, lila Verläufe). Mit gutem Prompting lässt sich das gut steuern.

Workflow — so entsteht eine Website

Briefing 15–30 Min

Kundenanruf oder Fragebogen: Branche, Zielgruppe, Wettbewerber, gewünschte Farben, Ziele der Website.

KI-Texte generieren 20–40 Min

Beispiel-Prompt: "Schreibe Texte für die Website eines Berliner Klempners. Zielgruppe: Hausbesitzer 30–60. Ton: vertrauenswürdig, lokal, kompetent. Benötigt: Hero-Text, 5 Vorteile, Über-uns-Text, CTA-Texte."

Design erstellen 60–120 Min

Builder öffnen, Template wählen, Texte einfügen, Bilder mit KI generieren (Midjourney/DALL-E), Farben & Logo anpassen.

Review & Feedback 30 Min

Vorschau-Link an Kunden schicken, Änderungswünsche einarbeiten — 1–2 Runden einplanen.

Launch & Übergabe 30 Min

Domain verbinden (Kunde kauft, ihr helft), live schalten, kurze Einweisung per Loom-Video.

Gesamtzeit pro Website: 3–5 Stunden.

Häufige Fehler — was ihr vermeiden solltet

Zu billig anfangen

Unter 500 € macht ihr euch selbst kaputt. Eure Zeit hat einen Wert — und der Markt akzeptiert höhere Preise als ihr denkt.

Keine Anzahlung verlangen

Immer 50 % upfront. Kunden, die das nicht bezahlen wollen, zahlen am Ende oft auch den Rest nicht.

Endlos Änderungen machen

Klare Änderungsrunden im Vertrag festlegen. Danach wird jede weitere Runde stundenweise abgerechnet.

Alles alleine machen wollen

KI übernimmt 80 % der Arbeit. Wer das nicht nutzt, verschenkt seinen Hauptvorteil.

Keine Pflegeverträge anbieten

Der sicherste Einkommensstrom überhaupt. Schon 15 Verträge à 99 € ergeben ~1.500 € pro Monat — bevor ihr morgens aufsteht.
